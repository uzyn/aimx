Why an inbox of its own

An agent that lives behind your Gmail account is a tenant in someone else's tenant in someone else's house. Every reply, every attachment, every check-in goes through three layers of OAuth and rate-limiting before it reaches your code. With its own MX record, the agent has an address you point to, an inbox it owns, and a delivery path it controls.

Mail on disk is mail you can grep. Markdown with TOML frontmatter is mail an LLM reads without a parser. Six months of correspondence become a find invocation, not an IMAP client. Email is the original cross-org protocol; Markdown is the format LLMs were trained on. Pair them and you have an agent-native medium.

Pushing beats polling. The daemon fires on_receive the moment SMTP DATA completes — no long-poll loops, no glue between SMTP and the agent. The MCP server exposes twelve typed tools the agent calls directly: list, read, send, reply, mark-read, mailbox CRUD.

What it is

AIMX is a self-hosted mail server for AI agents. One binary. One aimx setup command and your agents have their own email addresses on a domain you control. No Gmail, no OAuth, no SaaS.

Mail arrives over SMTP, gets parsed into Markdown with TOML frontmatter, and lands on disk. Outbound mail is DKIM-signed and delivered direct to the recipient's MX. Agents read and send through the built-in stdio MCP server, the aimx CLI, or the filesystem.

Works with any MCP-capable agent — Claude Code, Codex CLI, OpenCode, Gemini CLI, Goose, OpenClaw, Hermes, NanoClaw.

Where mail lives

Every message — inbound and outbound — is a Markdown file on your disk. Frontmatter is TOML. The body is the body. No .eml, no MIME parser, no database to back up.

Read it with cat. Tail it with an agent. Grep across a year of correspondence in a single command.

Quick start

Terminal
$curl -fsSL https://aimx.email/install.sh | sh$sudo aimx setup

You need sudo — port 25 is privileged. A VPS with port 25 open (inbound and outbound) and a domain you control are the only requirements. Not sure your VPS allows port 25? Check first, no install:

Terminal
$curl -fsSL https://aimx.email/portcheck.sh | sh

portcheck.sh is a thin alias for install.sh --port-check-only; both URLs run the same checks. Want to see the scripts before running them? Read install.sh or Read portcheck.sh.

What’s in the box

  • Single binary. Written in Rust. No runtime dependencies, no database, no relay.
  • Direct MTA-to-MTA. Outbound goes straight to the recipient's MX. Inbound hits your server on port 25. Feels like an API call.
  • Push, not poll. Inbound fires on_receive hooks the moment SMTP DATA completes. Outbound fires after_send when the MX attempt resolves. No long-poll loops, no glue between SMTP and the agent — just MCP tool calls and event fires.
  • Markdown emails. No .eml, no database. Just Markdown with TOML frontmatter — LLM- and RAG-friendly without a parser.
  • Built-in MCP. Stdio MCP server ships with the binary. Twelve tools covering mailbox CRUD, read, list, send, reply, mark-read.
  • You own the inbox. Mail lives on your disk, under your domain. Nothing phones home. MIT licensed.

FAQ

Do I need a VPS for this?

Yes. AIMX binds port 25, which is privileged and almost always blocked on managed PaaS. A small VPS with port 25 unblocked (Hetzner, OVH, others — see the Getting Started chapter) is the minimum.

Why not just use the Gmail API or SendGrid?

Both work for outbound. Neither gives you inbound mail as files on disk for an agent to read, hook on, and reply to. AIMX is symmetric: send and receive are first-class.

Is this safe to run for production agents?

It’s prerelease. The format is stable; the surface area is small (single binary, one daemon, no DB). Read the Security chapter before pointing real workloads at it.

Does AIMX phone home?

The binary doesn’t. No telemetry, no callouts. (This site uses Google Analytics for traffic measurement and Google Fonts for type — that’s separate from the binary.)

How is mail stored?

Markdown files with TOML frontmatter, one file per message, partitioned by mailbox. cat, grep, rsync, and git all work on it.
